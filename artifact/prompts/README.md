# Prompts and Planning Artifacts

This directory contains the actual prompts, planning documents, and reference
materials the practitioner used to drive the agentic-coding workflow described
in the paper. They are presented here as evidence of what "Phase 1: Contextual
Grounding" and "Phase 2: Collaborative Specification" produced in practice.

## Inventory

### `claude-md.md`

Verbatim copy of the project-root `CLAUDE.md` from the hackathon repository.
This file functions as the **agent contract**: every Claude Code session
loads it as system context. It encodes:

- Tech-stack decisions (Bun, Vite, React, TypeScript, Tailwind, Zustand)
- Project structure and architectural conventions
- Demo flow (the 3-minute scenario the agents must converge on)
- Style guide (color palette, typography, component classes — concrete,
  not aspirational)
- Agent task assignments (how work was decomposed across roles)
- Key design principles (teacher empowerment, bounded worlds, Socratic AI,
  process visibility)
- Beads workflow integration (issue tracker conventions)
- bv (beads-viewer) usage for triage analysis

The size and density of this file (≈19 KB, ≈380 lines) is itself a data
point: it is the **externalized tacit knowledge** the paper identifies as
necessary to align agents with practitioner intent. Compare against typical
README-only projects, which assume tacit context the agent does not have.

### `refs/`

Reference materials curated during Phase 1 — the "outside context" the
agent needed in order to make aligned product decisions. These are
**not** generated by the practitioner; they are existing documents
gathered, converted to markdown, and placed where Claude Code could read
them.

| File | Purpose |
|---|---|
| `event-description.md` | Hackathon event context (Hacks/Hackers + Atlantic + Infactory) |
| `ideas-from-the-atlantic.md` | The Atlantic's product team challenge prompts |
| `verge-interview-with-the-atlantic-ceo.md` | Domain context: Atlantic's AI strategy, "enshittification" concerns, subscription model |
| `competitor-ideas.md` | Other 12 teams' pitches — used for competitive differentiation (anonymized for this artifact) |

#### Anonymization note

`competitor-ideas.md` originally contained participant names, LinkedIn
URLs, email addresses, and external Google Doc links from other
hackathon participants. For this artifact bundle the names and personal
links have been replaced with team identifiers (`[Team 1]` through
`[Team 10]`) and brief role descriptors. The technical content of each
pitch is preserved verbatim so the artifact can support the paper's
12-team variation discussion (Section 4 of the camera-ready). The
practitioner whose workflow is the case-study subject (Andrew Zigler)
is publicly named in the paper itself; he is not represented in
this competitor table because his pitch was not pre-published in the
shared document.

## What's deliberately **not** here

- **Source code** (`src/`). This artifact is about the *preparation*
  that preceded code, not the code itself. The hackathon repository
  contains 8,496 lines of TypeScript; reproducing it in the artifact
  would obscure the paper's central claim that preparation is what
  matters.
- **`.env.local`** and any credentials. The Infactory API key has been
  rotated; even so, secrets do not belong in published artifacts.
- **Per-message session transcripts.** The Claude Code session JSONLs
  total ≈30 MB and contain incidental detail (tool-call traces, file
  reads, intermediate failures) without consistent narrative value.
  The aggregate metrics extracted from them are summarized in
  `../data/session-analysis.md`.

## How to read this directory

The intended reading order, if you are trying to reproduce the workflow:

1. **`refs/event-description.md`** and **`refs/ideas-from-the-atlantic.md`** —
   the situation that triggered the work
2. **`refs/verge-interview-with-the-atlantic-ceo.md`** — domain context
3. **`refs/competitor-ideas.md`** — competitive landscape
4. **`plans/idea.md`** — the product idea that emerged
5. **`plans/conversation-log.md`** — the brainstorming that refined it
6. **`plans/api-exploration.md`**, **`plans/api-info.md`**,
   **`plans/api-status.md`** — API understanding
7. **`plans/topic-research.md`**, **`plans/selected-data.md`** — content
   selection
8. **`plans/ui-explainer.md`** — UI specification
9. **`plans/simulated-progress.md`** — demo state plan
10. **`claude-md.md`** — the consolidated agent contract that ties it all
    together
11. (Then the agents started.)

This ordering is the actual chronological sequence in which these
documents were produced during the ~17-hour preparation phase. The
hackathon's 64 beads were dispatched **after** all of the above existed.
