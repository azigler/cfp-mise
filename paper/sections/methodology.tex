\section{The Mise en Place Methodology}
\label{sec:methodology}

We propose \emph{mise en place} as a structured preparation methodology for agentic coding.
The term, borrowed from professional culinary practice, describes the discipline of
arranging every ingredient and tool before cooking begins. In a professional kitchen,
the speed of service is a direct function of the thoroughness of preparation: chefs
who invest time in \emph{mise en place} execute complex dishes fluidly, while those
who skip preparation spend service time searching, measuring, and recovering from
avoidable errors. We observe an analogous dynamic in AI-assisted software development.
When practitioners invest in deliberate preparation---externalizing domain knowledge,
producing detailed specifications, and decomposing work into structured task
records---the resulting agent execution is faster, more aligned with intent, and
requires less corrective iteration.

The methodology formalizes this preparation into three sequential phases, each
producing defined artifacts that feed into the next. The phases are designed to
be completed before any implementation begins, following the backward design
principle~\cite{wiggins1998understanding}: start with desired outcomes and work
backward to the activities that will produce them. Figure~\ref{fig:methodology}
illustrates the three phases and their relationship to agent execution.

\subsection{Phase 1: Contextual Grounding}

The first phase externalizes domain expertise, tacit knowledge, and value
judgments into structured documents that agents can consume. Its purpose is to
capture the context that agents cannot generate independently---what
Polanyi~\cite{polanyi1966tacit} characterized as tacit knowledge, the
understanding that practitioners possess but struggle to articulate. In agentic
coding, this includes institutional context, domain-specific constraints,
competitive landscape awareness, design philosophy, and the practitioner's
accumulated judgment about what constitutes quality in the target domain.

The artifacts produced during contextual grounding are briefing documents:
markdown files that encode domain knowledge, competitive analysis, stakeholder
concerns, architectural preferences, and design philosophy. These documents
serve a dual purpose. First, they force the practitioner to articulate knowledge
that would otherwise remain implicit, surfacing assumptions and priorities that
might never reach the agent through prompts alone. Second, they create a
persistent context layer that agents can reference throughout implementation,
reducing the need for repeated clarification.

This phase connects directly to backward design~\cite{wiggins1998understanding}.
Rather than beginning with implementation (the ``what''), contextual grounding
begins with outcomes (the ``why''): what problem does the software solve, for
whom, and what constitutes success? By anchoring preparation in desired outcomes
rather than technical features, the practitioner establishes alignment criteria
that propagate through subsequent phases.

\subsection{Phase 2: Collaborative Specification}

The second phase produces detailed design artifacts through structured
human-agent dialogue. The practitioner describes intent; the agent proposes
implementation details; the practitioner accepts, rejects, or modifies the
proposal. This iterative refinement continues until both parties converge on a
specification that is detailed enough for implementation without further
clarification.

The key mechanism in this phase is the encoding of value judgments as
specification constraints. What to \emph{exclude} is as important as what to
include. Scope management---the deliberate decision to constrain features,
simplify interactions, or defer capabilities---represents a class of judgment
that agents cannot make independently, because it requires understanding the
practitioner's priorities, the deployment context, and the audience. During
collaborative specification, these judgments are surfaced, debated, and locked
into the specification as binding constraints.

The output is a comprehensive design document that specifies every screen,
interaction, data flow, and quality standard. Crucially, this document captures
not just the \emph{what} but the \emph{why} behind each decision, enabling
agents to make aligned micro-decisions during implementation without escalating
to the practitioner. This distinction separates collaborative specification
from conventional requirements gathering: the specification encodes rationale,
not just requirements.

\subsection{Phase 3: Task Decomposition}

The final preparatory phase converts the specification into structured,
dependency-aware task records. We use a system called
Beads~\cite{yegge2025beads, emanuel2025beadsrust}---lightweight, Git-backed
JSON records with priorities, dependencies, and acceptance criteria---though
the principle applies to any structured task tracking system that supports
dependency declaration.

Fine-grained decomposition serves a specific architectural purpose: it enables
parallel agent execution without coordination overhead. When each task is
independently executable---with clear boundaries, defined inputs and outputs,
and explicit dependency relationships---multiple agents can work simultaneously
on different parts of the system. The coordination burden shifts from runtime
(agents negotiating with each other) to preparation time (the practitioner
defining task boundaries), where human judgment about system architecture is
most valuable.

The output of this phase is a dependency graph where each node represents an
independently executable unit of work. The graph structure enables the
practitioner to identify the critical path, allocate agents to parallelizable
tasks, and monitor progress through structured completion signals rather than
ad hoc status checks. This decomposition also provides a natural mechanism for
prioritization: core features receive high priority and are executed first,
while stretch goals and polish items are explicitly deferred, preventing scope
creep during implementation.

\subsection{Relationship to Existing Approaches}

The three phases of mise en place are not individually novel. Contextual
grounding draws on Polanyi's framework for tacit knowledge
externalization~\cite{polanyi1958personal}. Collaborative specification
resembles aspects of the RPI (Research, Plan, Implement)
methodology~\cite{horthy2025rpi} and specification-driven
development~\cite{github2025speckit}. Task decomposition echoes Huntley's
Ralph Loop~\cite{huntley2025ralph}, which advocates breaking problems into
the smallest possible units for agent iteration.

What mise en place contributes is the integration of these concerns into a
sequential, phase-gated methodology with defined artifacts and a clear
theoretical grounding in backward design. The methodology's contribution is
not any single phase but the discipline of completing all three before
implementation begins---the insistence that preparation is not overhead to
be minimized but the primary determinant of implementation quality.
